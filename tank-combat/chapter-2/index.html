This is chapter two of an interactive book about data-oriented design. Chapter 1 was two tanks on a single grid of walls. Chapter 2 grows the world to a 4×4 arrangement of screen-grids and adds pathing: a tank you give a destination, and it routes itself there along the shortest wall-free way, across screens and around the world's wrap. As before, every data structure below is live — it reads and pokes the running game's memory directly.

The whole chapter is one idea applied twice: the path is a table, and a moving tank just does a lookup. Working out the shortest route is the expensive part, so we do it once against the rarely-changing walls and store the answer; per tick a tank reads the next step. The art is making the stored answer small.

The world is one big toroidal grid, organised as screens

Mechanically there is a single (4·20)×(4·15) = 80×60 toroidal grid. The 4×4 split into 20×15 screens is an organisation — for display (the viewport shows one screen), for scrolling, and for the two-level pathing below. A tank crosses from one screen into its neighbour wherever the border cell and the neighbouring cell are both open — exactly chapter 1's "depends only on the wall on the far side" rule, now applied at every inner screen edge and at the outer wrap. The default map leaves a one-cell gap at the centre of each screen border; those gaps line up, so the whole world is connected. Scroll around with the arrows; the little map shows where you are:

the 4×4 world · the bordered square is the viewport · dots are tanks · click a screen to jump there

The viewport is presentation only: scrolling changes what you see, never the simulation. A tank keeps pathing whether or not its screen is in view.

Two tank classes, stored as a partition — no class tag

There is one manual tank (amber, index 0), driven exactly as in chapter 1, and three pathed tanks (indices 1–3) that route themselves. Both classes share every per-tank field and run through the same turn/move transforms — the only difference is where each tank's input comes from: the manual tank from the keyboard/pad, a pathed tank from the path lookup. So there is no class byte at all; the index range is the class (0 manual, 1+ pathed). A pathed tank's per-tick input is just the next cardinal toward its goal, turned into the same button bits the manual tank uses and fed into the same movement model — the movement code is not forked.

per-tank data · class is the index · for pathed tanks: goal cell, status, and the current direction lookup

Level 1 — within a screen: all-pairs, stored as distance

Inside one screen you must be able to route any cell to any cell. That is the all-pairs table the grid-paths project builds. It sounds huge — 300 cells is ~90,000 ordered pairs — but it shrinks once you store it to its real domain, and keeping it small is the exercise.

The key move: store the metric, derive the arrow. The distance is symmetric (dist(a,b) == dist(b,a)), so we keep just the upper triangle — one byte per unordered pair. The next direction is not symmetric, so we don't store it; we read it off the distances at runtime: the next step from a cell toward a target is the neighbour whose stored distance to the target is one less. One table serves both the per-tick arrow and the Level-2 weights below. Pick a target cell and watch its flow field — the arrow every cell would take to reach it, derived live from the distances:

a screen's flow field · choose a screen and click a target cell · arrows are derived from the stored distances

How big is one entry? A distance here is small, bounded data — not an arbitrary number — so we size to what it can actually be. A shortest path inside a screen never revisits a cell, and no 2×2 block can sit entirely on it (four cells would close a loop), so the longest in-screen distance is at most 300 − 10·7 − 1 = 229: it provably fits one byte (255 is reserved for "unreachable"). So we store a uint8, not two — 45,150 pairs × 1 byte = 45 KB/screen, 706 KB for all 16 — and the whole chapter stays inside chapter 1's 1 MiB, still with no dynamic allocation. And because the table is built from the grid (at build/edit time, not from arbitrary input), we also just measure the longest distance — shown in the counters above — so if a screen ever did exceed the byte, we'd see it and fix the design rather than pay for two bytes everywhere on the chance. Built by a breadth-first fill on init, rebuilt for one screen when you edit a wall there — rarely — and read by a lookup every tick. Level 2 — between screens: a next-hop matrix over edge points

The connecting edge points are the open border cells whose neighbour across the border is also open — the matching gaps. Over just those points we build a second all-pairs table: for a start point A and an end point B, the next edge point on the shortest route (and its length). The win is that this is built on top of Level 1: two edge points on the same screen are joined by an edge weighted with that screen's Level-1 distance; crossing a matched border costs 1. So Level 2 never re-walks the grid — it reuses the distances Level 1 already stored, and a small Floyd–Warshall fills the matrix. With the default map there are only 64 edge points, so the matrix is tiny.

the connecting edge points (matched gaps) and the next-hop matrix · pick a source edge point to see its row

Composing the two levels into a route

When you click a destination, we fold the matrix into a tiny per-tank vector: for every edge point, the shortest remaining distance to the goal (Level-2 distance to a goal-screen entry point, plus that screen's Level-1 distance in to the goal cell). That fold happens once, when the destination changes — not per tick. After that, each tick a pathed tank in screen S either follows S's Level-1 flow straight to the goal (if it's there and reachable) or picks the exit edge point that minimises distance-to-the-exit + 1 + remaining-from-its-partner — a handful of table reads, never a search. Because every step strictly decreases the true remaining distance, the route is the globally shortest one, it never enters a wall, and an unreachable destination produces no path at all (shown as such) rather than a spin. The highlighted dots are this route, read straight out of the same tables the tank follows.

What this bought us

No allocation, no floats, no objects — a 4×4 world with self-routing tanks in a few kilobytes of WebAssembly plus a couple of megabytes of flat, packed integer tables, all sized to their real domain and all printable on this page. Each table has a job you can state: how often it is built, how often it is read, and why it is shaped the way it is. The simulation is still fully separated from rendering and wasm, so the whole thing — the pathing included — is tested natively with no browser or GPU.
